Freeline性能测试

基本说明

Freeline是由蚂蚁聚宝Android团队开发的一款针对Android平台的增量编译工具。它可以充分利用缓存文件，在几秒钟内迅速地对代码的改动进行编译并部署到设备上，有效地减少了日常开发中的大量重新编译与安装的耗时。

Freeline能够为Android开发者节省很多喝杯咖啡的时间 : )

Freeline将整个编译过程拆分成多个小模块，根据模块间的依赖关系并发进行编译任务。它可以对所有编译缓存文件做到充分地利用，真正地做到了“增量”编译。同时，它也提取了buck的部分组件（dx与DexMerger），来加速构建流程。Freeline会在app运行时启动一条独立进程来运行Socket Server以进行开发机与app的通信，同时也能确保即使在app出现crash之后，仍然能够使增量编译持续生效。

Freeline是通过multi-dex的方案来实现Java代码的增量更新的。与此同时，我们也专门对aapt工具做了深度优化，给出了独立的FreelineAapt资源打包工具用来生成增量的资源包以及加快资源包的打包速度。运行期的资源替换，Freeline中使用了Instant-Run的方案。除此之外，Freeline也支持运行期动态更新so文件。

Service要注意的几点

  • 服务运行在主线程。
  • startService()系统将调用onStartCommand()，通过stopService()或stopSelf()停止服务。
  • 如果选择实现 onStartCommand() 回调方法，则必须显式停止服务，因为系统现在已将服务视为已启动。在此情况下，服务将一直运行到其通过 stopSelf() 自行停止，或其他组件调用 stopService() 为止，无论其是否绑定到任何客户端。
  • bindService()系统将调用onBind()，多个组件可以绑定同一service，全部取消绑定unbindService()后，该服务即会被销毁。
  • 绑定是异步的。bindService() 会立即返回，“绝对不会”使 IBinder 返回客户端。要接收 IBinder，客户端必须创建一个 ServiceConnection 实例，并将其传递给 bindService()。ServiceConnection 包括一个回调方法，系统通过调用它来传递 IBinder。
  • 如果服务同时具有已启动和绑定状态，则服务启动后，只有所有绑定都取消后，再调用stopService()或stopSelf()才能真正停止服务。
  • 如果您的服务已启动并接受绑定，则当系统调用onUnbind()方法时，如果想在客户端下一次绑定到服务时接收onRebind()调用（而不是接收onBind()调用），则可选择返回 true。onRebind() 返回空值，但客户端仍在其 onServiceConnected() 回调中接收 IBinder。
  • android:exported = "false"服务仅用于当前应用，其他应用无法启动。

Android.mk文件

  • Android.mk是一个GNU makefile片段，编译器会解析其一次或多次。

最简单的Android.mk

# Android.mk必须由定义LOCAL_PATH变量开始
# my-dir是编译系统提供的宏，返回当前文件所在文件夹路径
LOCAL_PATH := $(call my-dir)

# 重置几乎所有的LOCAL_XXX变量。必须在每个module描述前声明该句。
include $(CLEAR_VARS)

# 指定生成的文件名。
LOCAL_MODULE := hello-jni

# 指定源文件。（多个源文件时，空格隔开）
LOCAL_SRC_FILES := hello-jni.c

# 指定编译行为。这里是生成动态库.so
include $(BUILD_SHARED_LIBRARY)

宏和变量

如果需要定义自己的变量，可以考虑使用MY_开头，而不是LOCAL_、PRIVATE_、NDK_、APP，因为编译系统内部有使用这些开头的变量。

NDK开发要注意的几点

  • 如果两个库分别实现了相同的签名方法，则会导致链接错误。在C中，“签名”只意味着方法名。在C++中，“签名”不仅包括方法名，还包括参数类型、数量和顺序。
  • SDK提供了NativeActivity辅助类，可以让我们写一个完全基于native的应用。NativeActivity处理Android框架与native code之间的通信，因此，你不需要继承它或者调用它的方法，你只需要在AndroidManifest.xml中定义你的应用为native的即可。
  • 使用NativeActivity的应用仍然运行在虚拟机中。
  • 两种实现native activity的方法：
    • native_activity.h包含你创建native activity所需的回调接口和数据结构。由于是主线程处理回调，所以回调阻塞将导致ANR。
    • android_nativity_app_glue.h基于native_activity.h接口定义了一个静态辅助类库，其使用非主线程处理回调、输入输出等事件，以防止主线程阻塞。 你可以修改<ndk_root>/sources/android/native_app_glue/android_native_app_glue.c的实现。

Android任务和返回栈

注：“任务”是指在执行特定作业时与用户交互的一系列 Activity。 这些 Activity 按照各自的打开顺序排列在堆栈（即“返回栈”）中。

launchMode 属性的四种启动模式

  • standard
    • 默认模式。
    • Activity 可以多次实例化。
    • 系统在启动 Activity 的任务中创建 Activity 的新实例并向其传送 Intent。
  • singleTop
    • Activity 可以多次实例化。
    • 如果当前任务的顶部已存在 Activity 的一个实例，则系统会通过调用该实例的 onNewIntent() 方法向其传送 Intent，而不是创建 Activity 的新实例。
    • 如果当前任务的已存在 Activity 的一个实例，但不在顶部，则会向堆栈添加新实例。
    • 当 Activity 的现有实例处理新 Intent 时，则在新 Intent 到达 onNewIntent() 之前，用户无法按“返回”按钮返回到 Activity 的状态。
